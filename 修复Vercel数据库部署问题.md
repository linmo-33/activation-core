# Project Overview

这是一个基于 Next.js 的激活码管理系统，使用 Prisma ORM 和 SQLite 数据库，包含管理员认证、激活码生成验证、日志记录等功能。项目当前在本地开发环境中正常运行，但部署到 Vercel 后出现数据库连接错误。

# Analysis (Populated by RESEARCH mode)

## 核心问题诊断

通过深入分析项目结构和配置，确定了根本问题：

### 1. 数据库配置不兼容 serverless 环境

- **当前配置**: Prisma schema 使用 SQLite 数据库 (`provider = "sqlite"`)
- **DATABASE_URL**: 设置为相对路径 `"file:./dev.db"`
- **问题**: SQLite 文件数据库在 Vercel serverless 环境中无法正常工作

### 2. Vercel 平台限制分析

- **文件系统限制**: Vercel serverless 函数运行在只读文件系统中（除/tmp 目录外）
- **容器隔离**: 每次函数调用可能在不同容器中执行，无法保证数据持久化
- **构建问题**: package.json 中的构建脚本包含`prisma db push`，在 serverless 环境中无法执行

### 3. 错误流程追踪

- 用户访问管理员登录页面 (`/admin/login`)
- 提交登录表单，调用 `/api/admin/login` API
- API 尝试通过`prisma.admin.findUnique()`查询管理员账户
- Prisma 无法连接到 SQLite 数据库文件，抛出数据库连接错误
- 返回"找不到数据库"错误信息

### 4. 关键文件分析

- **登录 API** (`src/app/api/admin/login/route.ts`): 依赖数据库查询管理员账户
- **数据库连接** (`src/lib/db.ts`): 标准 Prisma 客户端配置，无 serverless 优化
- **初始化脚本** (`scripts/init-admin.ts`): 需要数据库写入权限，在 Vercel 上无法执行

### 5. 环境差异

- **开发环境**: SQLite 文件存储在本地文件系统，可读写访问
- **生产环境(Vercel)**: 文件系统只读，SQLite 无法创建或访问数据库文件

## 技术依赖关系

- Next.js 15.4.1 应用框架
- Prisma ORM 数据库管理
- SQLite 数据库引擎（问题源头）
- JWT 认证机制
- bcryptjs 密码加密
- 管理员账户系统依赖数据库

# Proposed Solution (Populated by INNOVATE mode)

## 解决方案转向：Docker 容器化部署

基于用户需求，从 Vercel serverless 部署转换为 Docker 容器化部署，保持现有 SQLite 数据库配置。

### 推荐方案：多阶段构建 + Docker Compose 编排

**核心优势**：

- 保持现有 SQLite 数据库，无需迁移数据库类型
- 完全可控的运行环境，避免 serverless 限制
- 支持数据持久化，容器重启不丢失数据
- 易于在任何支持 Docker 的服务器上部署
- 为未来功能扩展（Redis、反向代理等）预留空间

### 技术实施要点

**1. Dockerfile 设计**

- 多阶段构建：构建阶段安装依赖并构建应用，运行阶段使用轻量基础镜像
- Node.js 生产镜像优化，减少镜像体积
- 非 root 用户运行，提升安全性
- 健康检查配置，确保应用可用性

**2. Docker Compose 配置**

- 服务编排管理应用生命周期
- 环境变量统一管理
- 数据卷持久化 SQLite 数据库
- 端口映射和网络配置
- 重启策略和资源限制

**3. 数据持久化策略**

- 使用命名卷持久化 SQLite 数据库文件
- 挂载点：`/app/prisma/dev.db`
- 确保数据在容器重新创建后仍然保留

**4. 环境配置优化**

- 生产环境变量配置
- 数据库初始化脚本集成
- JWT 密钥和管理员账户安全配置

### 部署优势分析

**相比 Vercel 部署的改进**：

- ✅ 完全兼容 SQLite 数据库
- ✅ 无 serverless 冷启动问题
- ✅ 完整文件系统访问权限
- ✅ 可执行数据库初始化脚本
- ✅ 自定义服务器配置能力
- ✅ 成本可控，避免按调用计费

**技术风险评估**：

- 需要服务器维护和监控
- 需要处理备份和灾难恢复
- 扩展性相比云平台有限制（可通过容器编排解决）

### 实施复杂度

- **低复杂度**：保持现有代码结构不变
- **中等学习成本**：需要基础 Docker 知识
- **高灵活性**：支持各种云服务商和私有服务器

## Docker 化部署详细技术规范

### 核心文件创建计划

**1. Dockerfile 设计规范**

- 文件路径：`./Dockerfile`
- 多阶段构建架构：
  - Stage 1 (deps): 安装生产依赖
  - Stage 2 (builder): 构建应用和生成 Prisma 客户端
  - Stage 3 (runner): 生产运行环境
- 基础镜像：node:18-alpine（安全性和体积优化）
- 用户配置：创建 nextjs 用户（UID 1001），非 root 运行
- 工作目录：/app
- 健康检查：HTTP 端点检查应用状态

**2. Docker Compose 服务编排**

- 文件路径：`./docker-compose.yml`
- 服务名称：activation-core-app
- 端口映射：主机 3000 -> 容器 3000
- 环境变量：.env.docker 文件加载
- 数据持久化：命名卷 activation-core-data
- 挂载配置：./data:/app/data (SQLite 数据库目录)
- 重启策略：unless-stopped
- 网络：activation-core-network

**3. 数据库初始化脚本**

- 文件路径：`./docker-entrypoint.sh`
- 功能职责：
  - 检查并创建数据库目录权限
  - 执行 prisma db push 初始化数据库
  - 运行管理员账户初始化脚本
  - 启动 Next.js 应用
- 执行权限：755
- 错误处理：数据库连接失败重试机制

**4. 环境配置模板**

- 文件路径：`./.env.docker`
- 配置项目：
  - DATABASE_URL="file:/app/data/prod.db"
  - NODE_ENV="production"
  - JWT_SECRET（生成随机 32 位密钥）
  - ADMIN_USERNAME="admin"
  - ADMIN_PASSWORD（可配置或自动生成）
  - PORT=3000

**5. 健康检查 API 端点**

- 路径：`./src/app/api/health/route.ts`
- 功能：检查应用和数据库连接状态
- 响应格式：JSON 格式状态信息
- 用途：Docker 健康检查和监控

**6. Docker 构建忽略规则**

- 文件路径：`./.dockerignore`
